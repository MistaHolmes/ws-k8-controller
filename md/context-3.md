# Stateful Autoscaling Evaluation – Complete Context (Experiment A, B1, B2-Instrumented)

## 1. Research Objective

This project evaluates the limitations of CPU-based Horizontal Pod Autoscaling (HPA) for stateful, long-lived connection workloads (WebSocket/MQTT-style systems).

The core hypothesis:

> CPU-driven autoscaling is insufficient for persistent connection workloads and may cause:
> - Replica oscillation
> - Premature connection termination
> - Reconnection amplification (reconnection storms)
> - Control-loop instability

We aim to demonstrate this experimentally using Kubernetes, Prometheus, and controlled cyclic load patterns.

---

## 2. System Architecture

### Cluster
- Local multi-node cluster using `kind`
- Fresh cluster created per experiment
- Metrics Server installed manually
- Prometheus deployed for metrics aggregation

### Workload
- Custom WebSocket server (Python + asyncio)
- Deployed via Kubernetes Deployment
- Exposes:
  - WebSocket service (port 8765)
  - Metrics endpoint (port 8080)

### Instrumented Metrics
Using `prometheus_client`:

- `active_connections` → Gauge
- `new_connections_total` → Counter

These allow measuring:
- System state (connections)
- Event flow (connection arrivals)

### Experiment B2-Instrumented (Primary Experiment)

This is the finalized, fully instrumented version.

Enhancements:

1. Proper Prometheus instrumentation:
   - Gauge: `active_connections`
   - Counter: `new_connections_total`

2. Prometheus collector loop:
   - sum(active_connections)
   - sum(rate(new_connections_total[5s]))

3. Cyclic load:
   - HIGH phase: 60s
   - LOW phase: 90s
   - Repeated for 5 cycles

4. HPA configuration:
   - Target CPU utilization: 60%
   - minReplicas: 2
   - maxReplicas: 15
   - Scale-down stabilization: 60s

## 4. Critical Debugging Discoveries

### 4.1 HPA Showing `<unknown>`

Observed:

```

cpu: <unknown>/60%

````

Meaning:

- Metrics server not providing CPU metrics
- Or CPU `requests` missing in Deployment
- HPA unable to compute utilization

Resolution:

Add to container spec:

```yaml
resources:
  requests:
    cpu: 100m
  limits:
    cpu: 500m
````

Without CPU requests, HPA cannot function.

---

### 4.2 Prometheus Collector Stability

Issues encountered:

* Background subshells exiting due to `set -e`
* Relative path errors
* JSON block misalignment

Resolution:

* Use absolute paths via `$PROJECT_ROOT`
* Disable `-e` inside subshell collector
* Enforce strict 3-line log format:

  1. timestamp
  2. active_connections JSON
  3. reconnect_rate JSON

---

### 4.3 Parsing Fragility

Original parser assumed fixed column indices for HPA output.

Issue:

* `kubectl get hpa` formatting changed (e.g., `cpu: 45%/60%`)
* Column-based parsing failed silently

Resolution:

* Robust parsing:

  * Search for `%` token
  * Extract last integer as replica count
  * Remove positional dependency

## 6. Phenomena Being Measured

### 6.1 Replica Oscillation

Repeated scaling up and down due to:

* CPU threshold crossings
* Cyclic workload

Measured via:

* replicas.csv

---

### 6.2 Premature Connection Termination

Observed as:

* Sudden drop in `active_connections`
* Coinciding with replica reduction

This demonstrates:

> CPU-based scale-down kills live connections.

---

### 6.3 Reconnection Storm

Measured via:

```
sum(rate(new_connections_total[5s]))
```

Expected behavior:

* Spike immediately after replica drop
* Clients reconnect
* CPU temporarily increases
* Positive feedback loop

This provides:

> Causal evidence of instability amplification.
